#TITLE: SQLite Resource
#author: Alejandro Garcia

* Goal: Expose Sqlite Databases inside the RESTLY Components framework.

* Description
 RESTLY_SQLITE is a RESTLY_RESOURCE that only works on URIs of the form "sqlite://" name of the database.
 It is also an EIFFELSTORABLE that works with sqlite databases.

 An example of interactions with the database is:

** Let's list all the tables in the database.

 #+BEGIN_EXAMPLE
 local
   db : RESTLY_SQLITE
   p : PERSON
 do
   create db.make_with_address ("sqlite://my_in_memory_database")
   -- let's print all the tables in the sqlite table.
   across db as table loop
      print ("The table name is:%N")
      print (@ table.key)
   end


 end
 #+END_EXAMPLE

** Assuming there is a table called Person then reading from it should generate new PERSON.

 #+BEGIN_EXAMPLE
 local
   db : RESTLY_SQLITE
   p : PERSON
 do
   create db.make_with_address ("sqlite://my_in_memory_database")
   p := db ["persons/1"]
   p.set_age (p.age + 1)
   db ["persons/1"] := p

 end
 #+END_EXAMPLE

** The difficult case is to create a new person without an id and then letting SQLI use an aute increment.

 #+BEGIN_EXAMPLE
 local
   db : RESTLY_SQLITE
   new_id : INTEGER
   p,new_person : PERSON

   request_id: STRING
 do

   create db.make_with_address ("sqlite://my_in_memory_database")
   create p.make_with ("this_is_name", 18)
   request_id := p.out
   db ["persons"].extend_with (p, request_id)
   new_id := db.extended_id (request_id)
   new_person := db ["persons/" + new_id.out]
   p ~ new_person -- p and new_person are equivalent except that new_person has the id of the field.

 end
 #+END_EXAMPLE

* Decision: RESTLY_TABLE_ORIGIN [V] composes an ABEL repository

** Why ABEL (answers open question 1) — VERIFIED 2026-07-11
   ABEL's relational backend reads PRE-EXISTING tables (not only its own object schema).
   Verified present in our install at Eiffel_25.12/unstable/library/persistency/abel/:

   - PS_SQLITE_RELATIONAL_REPOSITORY_FACTORY: make; set_database (path to the .db file); manage ({PERSON}, "id"); new_repository returns a PS_RELATIONAL_REPOSITORY.
   - Mapping is by convention, zero mapping code: class name (lowercase) = table name, attribute names = column names.
   - Managed types: manage (type, primary_key_attribute) makes ABEL handle the auto-increment key AND write the minted id back into the Eiffel object after insert.
   - The connector exposes stored_types (table/type names read from the existing schema).
     RESTLY_SQLITE's table iteration rides on that; no raw sqlite_master access needed.

   EiffelStore targets ODBC/MySQL-style backends and needs manual mapping, so ABEL still wins.

   Caveats:
   - FLAT CLASSES ONLY: V must not reference other persisted objects (name: STRING, age: INTEGER fine; address: ADDRESS not).
     This is a documented constraint on every V author.
   - Types found in the schema but NOT managed get value semantics (override_expanded_type — no identity, no update).
     Every type exposed through a RESTLY_TABLE_ORIGIN must be managed.
   - The library lives in unstable/ — the compile check below is still pending before committing.

** Why composition, not inheritance
   - Identity-by-inheritance is reserved in this repo for what a store IS — its PATH or URI (RESTLY_FILE inherits RESTLY_FILE_NODE).
     RESTLY_TABLE_ORIGIN is not "a PS_REPOSITORY"; it is a store BACKED BY one.
   - Inheriting PS_REPOSITORY would drag its whole public interface and invariants into RESTLY_TABLE_ORIGIN, destroying the tiny store contract (has_key / item / force / put / extend / remove) that RESTLY_PROTOCOL [K, V] defines and RESTLY_DIRECTORY exemplifies.
   - Our R/S generic convention already names this relation: conversion between Representation and Store happens AT THE BOUNDARY of the store (same role KEY_CONVERTER plays in the EWF bridge).
     ABEL is that boundary: V -> rows on write, rows -> V on read.

   So: serialization is explicitly part of this work, but it is ABEL's part.
   RESTLY_TABLE_ORIGIN's job is only to expose it through the RESTLY store shape.

** Skeleton
   Mirrors RESTLY_DIRECTORY (RESTLY_PROTOCOL [RESTLY_URI_PATH, STRING]); here keys are row ids, values are typed objects.
   ABEL feature names below are indicative — verify against the library.

** Equality: is_equal excludes the auto-increment id
   V carries its id as an attribute, for two reasons:
   - a detached V must be able to find its way back to the table that contains it (the key alone lives with the client otherwise);
   - ABEL's PS_QUERY filters on attributes, so an id attribute is what makes item (k) — query by key — implementable at all.

   With the id inside V, default field-by-field equality would make p ~ new_person False after an insert.
   So V redefines is_equal to compare every field EXCEPT the auto-increment id.
   That makes the `p ~ new_person` postcondition in the examples above hold as written (`=` stays reference identity; `~` becomes "same domain values").

   Caveats this buys:
   - the id-blind equality leaks into every comparison of V (has, sets, search): two distinct rows with equal fields compare equal;
   - hash_code must be kept consistent with the redefined is_equal — exclude the id there too.

** Consequence for RESTLY_SQLITE
   RESTLY_SQLITE mirrors RESTLY_DIRECTORY: it owns the connection, iterates table names (via the ABEL connector's stored_types — the schema as read from the existing database), and hands out tables.

   The precedent is RESTLY_DIRECTORY.node: it returns the deferred ancestor RESTLY_FILE_NODE, creating RESTLY_DIRECTORY or RESTLY_FILE itself, and the caller object-tests.
   That works because file-vs-directory is a CLOSED set of two non-generic classes the directory can `create` by name.
   RESTLY_TABLE_ORIGIN [V] is an OPEN set: the database cannot `create` a type it cannot name, and an object test can only narrow an object already created with that generic.

   So the pattern splits in two:
   - db's navigation/traversal hands out UNTYPED table nodes (table name + repository — a stored_types entry) for listing and `exists` only, mirroring RESTLY_FILE_NODE;
   - the CLIENT creates the typed store from one:

 #+BEGIN_EXAMPLE
 persons: RESTLY_TABLE_ORIGIN [PERSON]
 ...
 create persons.make_with_repository (db.repository)
 #+END_EXAMPLE

   This replaces the untyped db ["persons/1"] path keys from the examples above: one store, one key type, one value type.

* HTTP QUERY (RFC 10008): RESTLY_SEARCHABLE, predicate reads on any store

** Why a new verb
   RFC 10008 (Proposed Standard, June 2026) defines QUERY: a SAFE and IDEMPOTENT method whose request body carries the query — GET's semantics with POST's payload.
   The RESTLY store contract covers GET/PUT/POST/DELETE (item / force-put / extend / remove) but has no verb for "safe read with a predicate"; today that read leaks out as ad-hoc feature calls with no HTTP mapping.
   QUERY names it, and safety plus idempotency make it a QUERY in the CQS sense too: a pure function of the store's state, freely retryable and cacheable.

** Shape: RESTLY_SEARCHABLE [Q, V], the RESTLY_POSTABLE mold
   A small mixin, one deferred feature; Q is the query REPRESENTATION, V the value type, following the R/S convention: converting Q into whatever the backend executes happens at the store boundary.
   Named SEARCHABLE/search, not QUERYABLE/query: `search` avoids the noun "query" doubling as a feature name that takes a query, and SEARCH was the method's pre-RFC name anyway.
   Result type is ITERABLE [V]: the meet point of ABEL's query cursor and EiffelBase2's V_ITERATOR, without handing either live cursor to the client (see the EiffelBase2 caveat below).

 #+BEGIN_EXAMPLE
 note
 	description: "[
 		QUERY (RFC 10008) protocol: safe, idempotent predicate reads.
 		Q is the query representation; execution is the store's business.
 	]"

** Draft for hash-backed stores (EiffelBase2 binding)
   In-memory stores bind Q := PREDICATE [V] and execute by scanning the backing table.
   This generalizes EiffelBase2's cursor-given-a-key idiom ({V_MAP}.at_key returns a cursor positioned at a key) to cursor-given-a-predicate.

 #+BEGIN_EXAMPLE
 class
 	RESTLY_RESOURCE [K -> HASHABLE, V]

 inherit
 	RESTLY_SEARCHABLE [PREDICATE [V], V]

 feature -- REST verbs

 	search (a_query: PREDICATE [V]): ITERABLE [V]
 			-- All values satisfying `a_query`.
 		require
 			error_400_bad_request: True
 					-- TODO(owner): contract
 		do
 			-- scan table.new_cursor, extend matches into a fresh V_LINKED_LIST
 		end

 end
 #+END_EXAMPLE

   Why drain into a fresh list instead of returning the live V_ITERATOR: EiffelBase2 cursors register as observers on their container, and {V_TABLE}.put requires observers.is_empty — a live cursor leaked past the store boundary would freeze every PUT/DELETE until the client remembers to forget_iterator.
   Draining inside `search` kills the observer at the boundary; the client gets a snapshot, which also matches QUERY's cacheable-result-set semantics.

** Caveats
   - Q propagates through combinators: a cache or forwarder wrapping a searchable store can only forward `search` by carrying Q in its own generic signature, so a combinator is searchable exactly over its backend's Q (PREDICATE [V] in-memory, PS_CRITERION on ABEL).
   - Binding Q to PS_CRITERION leaks an ABEL type into RESTLY_TABLE_ORIGIN client signatures; acceptable for v1 because those clients already hold a PS_RELATIONAL_REPOSITORY to create the table.
     A RESTLY-owned query representation is deferred until a third binding forces one.
   - Accept-Query negotiation (which media types a store accepts) is HTTP surface, so it belongs to the EWF bridge, not to the store protocol.
   - RFC 10008's Location-for-results (a result set minted as its own sub-resource URI) fits the combinator style but is out of scope for v1.
