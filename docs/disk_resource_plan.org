* Context

RESTLY has one edge resource so far: ~RESTLY_HTTP_CLIENT~ (store over the wire). This adds the second scheme's resource --- a directory on disk --- realizing the ~RESTLY_DIRECTORY~ / ~RESTLY_FILE~ fragment of =scheme_handlers_design.org= without the registry/FILE_SCHEME machinery (same way the HTTP client was built before its scheme handler).

* Decisions

- ~RESTLY_DIRECTORY~ inherits ~RESTLY_PROTOCOL [RESTLY_URI_PATH, STRING]~ --- ~"[]"~ returns file contents only.
- Root: *strict* ~file:///~ URI, typed via the URI_TEMPLATE library (HASHABLE, validated structure) --- a new ~RESTLY_FILE_URI~ descendant, same pattern as ~RESTLY_URI_PATH~. String call sites still work through the convert clause. Being HASHABLE, it is also the future identity-map memo key (design doc Decision 1).
- Navigation: ~subdirectory alias "/" (a_segment): RESTLY_DIRECTORY~ (doc's ruling; the ~Disk["/sub/"].as_subdir~ shape was rejected by the doc's own trailing-slash ruling).
- Listing: ~entries: ITERABLE [RESTLY_FILE_NODE]~ view query --- NOT the ~RESTLY_LISTABLE~ mixin (it inherits ~RESTLY_PROTOCOL~, would force V=node -> incoherent ~force~; doc open problem 7).
- ~RESTLY_FILE~: plain CELL-shaped leaf (~item~/~put~/~exists~/~remove~), not protocol-conforming.
- V = STRING confirmed (bytes): writes decide it --- ~force~ of a handle is incoherent, and composition (~RESTLY_CACHE~, converters, ~github.cached_by (disk)~) needs V matching ~RESTLY_HTTP_CLIENT~. ~RESTLY_FILE~ appears only in navigation/traversal, never as protocol value.
- ~force~/~put~ with missing parent directory or key naming an existing subdirectory: *owner-precondition ~error_409_conflict~* (WebDAV RFC 4918 S9.7.1 precedent --- PUT does not create intermediate collections). No auto-mkdir.

Eager node list, lazy contents --- bytes read only at ~RESTLY_FILE.item~.

* Contract policy

All require/ensure/invariant clauses: placeholder ~True~ + ~-- TODO(owner): contract~ with suggested expression in comment, tags per the ~error_NNN_description~ grammar. ~item~/~remove~/~extend~ preconditions are inherited from ~RESTLY_PROTOCOL~ --- add nothing there.

* Recorded ceilings (v1, deliberate)

- *Path traversal:* ~dir ["/../x"]~ escapes the root --- a real trust boundary, parallel to ~RESTLY_URI_PATH~'s no-foreign-host rule. Surface as a prominent owner-contract suggestion (~not k.template.has_substring ("..")~ plus no ~{~/~?~) on the verb preconditions, not buried in ~file_system_path~.
- *Percent-encoding:* templates are opaque; ~%%20~ lands literally in filesystem paths. Spaces/Unicode filenames unsupported --- note in class description.
- *Node->key asymmetry:* ~entries~ nodes carry absolute paths; the protocol key for a node is reconstructed as ~"/" + node.name~. Acceptable; document it on ~entries~.
- *remove is file-only* (inherited ~has_key~ precondition excludes directories) --- deviation from the doc's kind-agnostic remove, deliberate for v1.

* Explicitly skipped (add when the scheme-handler layer lands)

FILE_SCHEME factory + identity map; ~create_subdirectory~ (MKCOL); typed ~file (name)~/~node (name)~ accessors; LISTABLE/traversable decoupling (doc open problem 7).
