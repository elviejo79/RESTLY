#+AUTHOR: Alejandro

* Context

Third edge resource, third URI scheme. ~RESTLY_HTTP_CLIENT~ is the store over
the wire; ~RESTLY_DIRECTORY~ is the store on disk. ~RESTLY_FTP~ is both at
once: a remote store whose keys and values are files --- structurally a
~RESTLY_DIRECTORY~ reached through a socket. Same pre-scheme-handler approach
as its two siblings: build the resource, defer the FTP_SCHEME factory.

* Decisions

- ~RESTLY_FTP~ inherits ~RESTLY_PROTOCOL [RESTLY_URI_PATH, STRING]~ +
  ~RESTLY_FTP_URI~ --- the client IS its root URL, same
  identity-by-inheritance pattern as ~RESTLY_HTTP_CLIENT~ (~set_template~
  hidden, ~make_from_uri_template~ redefined to re-initialize the session).
- ~RESTLY_FTP_URI~: clone of ~RESTLY_HTTP_URI~, invariant
  ~names_an_ftp_endpoint: template.starts_with ("ftp://")~ (owner contract).
- No new dependency: libcurl speaks FTP natively and is already in the tree
  via =libcurl_http_client.ecf=. Uses ~CURL_EASY_EXTERNALS~ directly (not
  ~HTTP_CLIENT_SESSION~; spike proved the session's DELETE is broken for FTP
  and response codes are incompatible). Verb -> FTP command mapping:
  | Verb                 | FTP command | libcurl mechanism                          |
  |----------------------+-------------+--------------------------------------------|
  | ~item~ (GET)         | RETR        | plain transfer on ~ftp://~ URL             |
  | ~force~/~put~/~extend~ | STOR      | ~CURLOPT_UPLOAD~ + ~READFUNCTION~          |
  | ~remove~             | DELE        | ~CURLOPT_QUOTE~ ("DELE <path>")            |
  | ~has_key~            | SIZE        | ~CURLOPT_NOBODY~ (no body transfer)        |
  | ~entries~ (deferred) | NLST        | dir URL (trailing ~/~) + ~DIRLISTONLY~     |
- V = STRING (bytes), matching both siblings --- required for composition
  (~RESTLY_CACHE~, converters, ~github.cached_by (ftp)~).
- Transfers are binary (TYPE I), libcurl's default --- never ASCII mode;
  the binary-roundtrip test pins this.
- Navigation: ~subpath alias "/" (a_segment): RESTLY_FTP~, same shape as
  ~RESTLY_HTTP_CLIENT.subpath~.

* Spike results (2026-07-08, resolved)

** Question
Does ~HTTP_CLIENT_SESSION~ (libcurl backend) pass ~ftp://~ URLs through?

** Live test (pyftpdlib on port 2123, curl CLI)

| Verb          | curl mechanism                  | FTP response code | Works?                                  |
|---------------+---------------------------------+-------------------+-----------------------------------------|
| GET existing  | default (~CURLOPT_HTTPGET~)     |               226 | Body correct; status ≠ 200              |
| GET missing   | default                         |               550 | curl rc=78                              |
| HEAD existing | ~-I~ (~CURLOPT_NOBODY~)         |               350 | Metadata returned (Content-Length, etc.) |
| HEAD missing  | ~-I~ (~CURLOPT_NOBODY~)         |               550 | Consistent with GET missing             |
| UPLOAD (STOR) | ~-T~ (~CURLOPT_UPLOAD~)         |               226 | Works                                   |
| DELETE        | ~-X DELETE~ (~CUSTOMREQUEST~)   |              (226) | *Broken*: sends RETR, not DELE          |
| DELE          | ~-Q "DELE"~ (~CURLOPT_QUOTE~)   |               250 | Works; but CURLOPT_QUOTE not in bindings|
| ~--include~   | ~CURLOPT_HEADER=1~              |               --- | Harmless for FTP; body stays clean      |

** Conclusions

1. *FTP response codes ≠ HTTP.* ~curlinfo_response_code~ returns FTP codes
   (226 success, 350 SIZE ok, 550 not found). The session's
   ~response_status_code~ returns these raw --- ~RESTLY_HTTP_CLIENT~ checks
   ~200 ~ response.status~ which fails for FTP's 226.
2. *DELETE is broken.* ~CURLOPT_CUSTOMREQUEST "DELETE"~ tells curl to use
   DELETE as the *request command* before transfer --- but that's an FTP
   ~RETR~ with "DELETE" sent as a preamble. FTP deletion requires
   ~CURLOPT_QUOTE "DELE <path>"~.
3. *~CURLOPT_QUOTE~ is not exposed* in the Eiffel ~CURL_OPT_CONSTANTS~ class
   (203 constants exposed, ~curlopt_quote~ is not among them).
4. *PUT with inline data* goes through the session's temp-file hack
   (=libcurl_http_client_session.e= line 140--159), ending in
   ~CURLOPT_UPLOAD~ --- should work for FTP STOR.
5. *~CURLOPT_HEADER=1~* is harmless for FTP --- body stays clean.

** Verdict: raw ~CURL_EASY_EXTERNALS~

All verbs go through ~CURL_EASY_EXTERNALS~ directly. ~RESTLY_FTP~ owns
~curl: CURL_EXTERNALS~ + ~curl_easy: CURL_EASY_EXTERNALS~ (same attributes
~LIBCURL_HTTP_CLIENT_SESSION~ has, exported to its request class).

Verb mapping via raw curl:
- ~item~ (RETR): ~CURLOPT_URL~ + default GET, read into ~CURL_STRING~.
  Success when ~curl_easy.perform~ returns ~curle_ok~.
- ~force~/~put~/~extend~ (STOR): ~CURLOPT_UPLOAD=1~ + read function from
  temp file (same hack the session uses). Success = ~curle_ok~.
- ~has_key~ (SIZE): ~CURLOPT_NOBODY=1~. File exists when ~perform~ returns
  ~curle_ok~; missing when it returns ~curle_remote_file_not_found~ (78).
- ~remove~ (DELE): need ~CURLOPT_QUOTE~ which is *not exposed*. Two options:
  a. Add ~curlopt_quote~ to a local ~CURL_OPT_CONSTANTS~ descendant (one
     external ~"return CURLOPT_QUOTE;"~) --- cleanest.
  b. Shell out to ~curl -Q "DELE <path>"~ for just this verb --- ugly but
     zero new C glue.
  Recommendation: option (a), one three-line class.

Status codes are not checked; success/failure determined by ~curle_ok~ vs
~curle_*~ error codes from ~perform~ (same way ~LIBCURL_HTTP_CLIENT_REQUEST~
could have done it).

   - inherits ~CURL_OPT_CONSTANTS~, adds ~curlopt_quote: INTEGER~ (one

3. =library/restly/src/resource/restly_ftp.e=

     (export ~{NONE} set_template~, redefine ~make_from_uri_template~)

     ~make_from_uri_template~
   - owns ~curl: CURL_EXTERNALS~ + ~curl_easy: CURL_EASY_EXTERNALS~
     directly (NOT ~HTTP_CLIENT_SESSION~)
   - each verb: init handle, set ~CURLOPT_URL~, set verb-specific options,
     perform, cleanup. URL for key ~k~ is
     ~template.to_string_8 + k.template~ (keys carry leading ~/~)
   - ~remove~: uses ~curlopt_quote~ via ~RESTLY_FTP_OPT_CONSTANTS~ to send

     suppress the (empty) transfer

   - ~item~: default GET; read body from ~CURL_STRING~
   - ~force~/~put~/~extend~: ~CURLOPT_UPLOAD=1~ + write data to temp file +
     read function (same approach as ~LIBCURL_HTTP_CLIENT_SESSION~)
   - ~subpath alias "/"~ --- suggested tag ~error_404_not_found~
4. =library/restly/testing/test_restly_ftp.e= --- see below

* Test strategy

**Server:** =pyftpdlib= (installed), anonymous with write access, one line:

#+BEGIN_SRC sh
python3 -m pyftpdlib --port 2121 --write --directory /tmp/restly_ftp_sandbox
#+END_SRC

Same contract as ~TEST_RESTLY_HTTP_CLIENT~: server must be running, stated
in the class note. No launch machinery in the tests (v1 ceiling below).

**Cross-check through RESTLY_DIRECTORY.** The FTP server serves a temp
directory that ~RESTLY_DIRECTORY~ can already read and write. The fixture
holds two views of the same sandbox:

#+BEGIN_SRC eiffel
ftp: RESTLY_FTP           -- ftp://localhost:2121
disk: RESTLY_DIRECTORY    -- file:///tmp/restly_ftp_sandbox/
#+END_SRC

- Seeding bypasses the protocol under test (HTTP-test philosophy): seed via
  ~disk~, exercise ~ftp~.
- Verification is independent: ~ftp.force~ then assert ~disk.has_key~ ---
  proves bytes crossed the wire, not a roundtrip through one code path.

Setup/teardown: sandbox recreation copied from ~TEST_RESTLY_DIRECTORY~
(~on_prepare~ recursive_delete + recursive_create_dir, ~on_clean~ delete).
Sandbox path is fixed (=/tmp/restly_ftp_sandbox=) because the externally
launched server is rooted there --- unlike the directory tests, no
per-class uniquifier.

* Recorded ceilings (v1, deliberate)

- *Server lifecycle is manual:* tests require a running pyftpdlib, same as
  the todobackend requirement for HTTP tests. Auto-spawn from ~on_prepare~
  when this gets annoying.
- *Anonymous only:* no credentials in v1. libcurl takes ~user:pass@host~
  URLs when needed; add a ~make_with_credentials~ then.
- *No ~entries~ in v1:* NLST parsing (line splitting, dir-vs-file probing)
  is real work; add with the traversal story, not before.
- *No FTPS/SFTP:* invariant admits only ~ftp://~. Widen the invariant when
  a use case shows up (FTPS is free in libcurl; SFTP is a different
  protocol entirely).
- *Status codes ignored on writes:* same ceiling ~RESTLY_HTTP_CLIENT~
  already has (its ~force~ discards the response too). Fix both together.
